# BEEGENIUS — план та поточний стан реалізації

## 1. Мета

Сайт про бджільництво "Bee Genius · Валентин" — блог, наукові статті, оголошення/пропозиції та проєкти (наприклад «З Бджолами по Життю»). Власна легка адмінка (без CMS), вхід через Google-акаунт. Дизайн спочатку відштовхувався від погодженого макета (`tmp/Бджоли_Валентина_автономна_версія.html`), далі розвивався окремо за запитами власника.

Це самостійний проєкт. Жодних C#/.NET компонентів тут немає.

## 2. Технологічний стек

- **Backend:** PHP 8.x, власний легкий front-controller (без фреймворку — `public/index.php`, маршрути через `switch(true)` + `preg_match`).
- **База даних:** MySQL/MariaDB (`beegenius`), доступ через PDO з підготовленими запитами всюди.
- **Файли:** фото, завантажені з адмінки — `public/uploads/articles` (поза git); фото самого дизайну — `public/assets/img` (в git); шрифти — `public/assets/fonts` (self-hosted Manrope, Fjordgard).
- **Frontend:** HTML/CSS (один файл `public/assets/css/styles.css`), без збірки/бандлера. Шрифти: **Manrope** (основний текст), **Petrona** (заголовки), **Fjordgard** — лише в логотипі шапки.
- **Редактор тексту:** Quill.js (з CDN) для тіла статей, із власною модалкою вибору/завантаження фото.
- **Автентифікація:** Google OAuth 2.0, без паролів у базі; білий список email у `config/config.php`.
- **Локальна розробка:** XAMPP (Apache + PHP + MySQL + phpMyAdmin), проєкт живе в `C:\xampp\htdocs\beegenius`.

## 3. Хостинг

**HQ10** (shared, cPanel): 25 ГБ SSD, до 10 сайтів, необмежений трафік, 10 MySQL БД, 1536 МБ RAM. Перенесення 3 існуючих WordPress-сайтів з VIP30 на HQ10 — окрема задача, не стосується цього проєкту.

## 4. Архітектура: код vs контент

```
/beegenius
  /app
    /controllers        ← маршрути/логіка публічних сторінок (по одному файлу на дію)
      /admin             ← контролери адмінки (CRUD статей/відгуків/проєктів/тегів/налаштувань)
    /views               ← HTML-шаблони (PHP), layout/header.php + layout/footer.php спільні
    /lib
      db.php             ← PDO-підключення
      helpers.php        ← формат дат, рендер тіла статті, site_url()/absolute_url(), seo_excerpt()
      auth.php           ← Google OAuth, is_admin()/require_admin()
      csrf.php           ← csrf_token()/verify_csrf()
      sanitize.php       ← sanitize_article_html() — вайтлист-санітайзер HTML з Quill
      tags.php           ← уся логіка тегів (sync/get/usage)
  /public                ← точка входу (index.php), assets (css/img/fonts) — в git
    /uploads/articles     ← фото від адмінки — В GITIGNORE
  /config                ← config.php (креденшли, поза git) + config.example.php (шаблон, в git)
  /db
    schema.sql           ← довідковий знімок актуальної схеми (не застосовується автоматично)
    seed.sql
    /migrations          ← послідовні пронумеровані .sql-файли, застосовуються вручну через phpMyAdmin
  database (MySQL)       ← статті, теги, проєкти, налаштування — не в git, окремі бекапи
```

**Деплой:** ще не автоматизований (заплановано: git pull/webhook, оновлює лише `/app` і `/public`; `public/uploads`, `/config` і БД деплой не чіпає ніколи).

## 5. Маршрутизація (реалізовано)

`public/index.php` — єдина точка входу, `BASE_PATH` обчислюється від `SCRIPT_NAME` (порожній на проді, `/beegenius/public` локально).

**Публічні:**
- `/` — головна (hero з фото, банер-каруселька оголошень, "чим я займаюсь", лічильник з опублікованого проєкту, відгуки, останні пости блогу)
- `/blog`, `/blog/{slug}` — блог: список (пагінація по 12, пошук за заголовком/тегом + фільтр тегів) і стаття
- `/science`, `/science/{slug}` — наукові статті: те саме + медичне застереження
- `/announcements`, `/announcements/{slug}` — оголошення/пропозиції: те саме
- `/projects`, `/projects/{slug}` — список проєктів (пагінація по 12) і сторінка одного проєкту
- `/z-bdzholamy-po-zhyttyu`, `/my-project`, `/cases` — 301-редиректи на конкретний проєкт (історичні адреси)
- `/sitemap.xml`, `/robots.txt` — генеруються динамічно з реальних опублікованих записів
- `/login`, `/auth/callback`, `/logout` — вхід/вихід через Google (з CSRF-захистом через `state`)
- `/dev-login` — заглушка входу лише для локальної розробки (`IS_LOCAL`, перевіряється і `SERVER_NAME`, і `REMOTE_ADDR`)

**Адмінка (усі — за `require_admin()`):**
- `/admin` — хаб: плитки з лічильниками по розділах
- `/admin/articles?type=blog|science|announcement` — список одного типу, з пагінацією (20/сторінку), пошуком/фільтром статусу/дат
- `/admin/articles/new`, `/admin/articles/edit/{id}`, `/admin/articles/save`, `/admin/articles/delete`
- `/admin/quotes`, `/admin/quotes/new|edit/{id}|save|delete` — список з пагінацією (20/сторінку)
- `/admin/projects`, `/admin/projects/new|edit/{id}|save|delete` — список з пагінацією (20/сторінку)
- `/admin/tags` (керування + перегляд використання, максимум 10 статей на тег у списку), `/admin/tags/delete`, `/admin/tags/list` (JSON для автопідказок)
- `/admin/home`, `/admin/home/save` — головна сторінка: тексти хіро/про мене, хіро-фото, і вбудований список карток "Чим я займаюся"
- `/admin/activities/new`, `/admin/activities/edit/{id}`, `/admin/activities/save`, `/admin/activities/delete` — картки "Чим я займаюся" (список — на `/admin/home`, старий `/admin/activities` редіректить туди ж)
- `/admin/settings`, `/admin/settings/save` — контакти
- `/admin/upload-image`, `/admin/media/list` — спільна бібліотека фото (Quill + обкладинки + хіро-фото)

## 6. База даних (реалізовано)

Таблиці (актуальна схема — `db/schema.sql`, застосовується покроково через `db/migrations/`):

- **articles** — id, `type` (`blog`/`science`/`announcement`), title, slug, excerpt, body (HTML з Quill, санітизований), cover_image, `is_banner`, `banner_from`, `banner_until` (банер-каруселька на головній), status (draft/published), published_at, created_at, updated_at.
- **tags** — id, name, slug (UNIQUE).
- **article_tags** — many-to-many articles↔tags, `ON DELETE CASCADE` на обидва боки.
- **projects** — id, sort_order, title, slug (UNIQUE, фіксується при створенні, не змінюється при редагуванні), subtitle, counter_value, counter_label, `show_on_home` (лічильник на головній підтягується саме звідси — лише один проєкт одночасно, і лише якщо опубліковано), description, cta_note, cover_image, status, created_at, updated_at.
- **quotes** — id, sort_order, quote_text, quote_year, quote_image.
- **site_settings** — key/value: тексти головної (`home_kicker/title/subtitle/about_title/about_text`), `home_hero_image` (фото хіро-банера), контакти (`contact_*`). Лічильник тут більше не зберігається — див. `projects.show_on_home`.
- **activities** — картки "Чим я займаюсь" на головній: id, sort_order, title, text.
- **admin_login_log** — email, success, ip, created_at.

Таблиця `categories` була видалена (замінена системою тегів).

## 7. Автентифікація та ролі (реалізовано)

- Вхід через Google OAuth 2.0, доступ до адмінки — лише email з `admin_whitelist` у `config/config.php`, перевірка жива на кожен запит (`is_admin()`), не кешується в сесії.
- OAuth-флоу захищений `state`-параметром (RFC 6749 §10.12) — генерується при редиректі на Google, звіряється на `/auth/callback` через `hash_equals()`, одноразовий.
- `require_admin()` — обов'язковий перший виклик у кожному контролері, що додає/редагує/видаляє дані (не лише при вході в `/admin`).
- CSRF-токени (`csrf_token()`/`verify_csrf()`) на всіх формах, що змінюють дані.
- Сесія: httpOnly + secure (на https) + SameSite=Lax; `session_regenerate_id()` при вході й виході (захист від session fixation).
- **Сесія завершується сама** (`app/lib/auth.php`, `start_user_session()`/`current_user()`): 30 хв бездіяльності або 8 год від входу в будь-якому разі — потім користувач розлогінюється, при наступному відкритті сайту бачить повідомлення "Сесію завершено через тривалу бездіяльність".
- `/dev-login` (заглушка для локального тестування) гейтиться і `SERVER_NAME`, і `REMOTE_ADDR` — саме лише IP не підробити заголовком `Host`.
- Ролі кількох редакторів (наприклад "може додавати контент, не може видаляти") — обговорено, свідомо відкладено до реальної потреби.

## 8. Функціонал адмінки

- **Хаб** (`/admin`): плитки-розділи з лічильниками замість однієї довгої сторінки.
- **Статті** (блог/наука/оголошення — одна форма, розрізняються полем `type`): Quill-редактор тіла, вибір/завантаження обкладинки через спільну модалку медіатеки, теги (чипси + автопідказки з наявних тегів, підтвердження створення нового тега), банер на головній із необов'язковим періодом показу (`is_banner` + `banner_from/until` — поля дат неактивні, поки не позначено чекбокс банера).
- **Теги:** окрема сторінка керування — список усіх тегів, кількість використань, до 10 повʼязаних статей (далі "…і ще N"); видалення тега прибирає його всюди (каскадно).
- **Проєкти:** CRUD, обкладинка, число-лічильник (необов'язкове) + прапорець "показувати цей лічильник на головній" (лише один проєкт одночасно, і лише поки опубліковано).
- **Відгуки:** CRUD, необов'язковий скріншот.
- **Головна сторінка** (`/admin/home`): хіро-фото (через медіатеку), тексти хіро й блоку "Чим я займаюся"; тут же — вбудований список карток "Чим я займаюся" з переходом на окремі форми додавання/редагування/видалення.
- **Контакти:** редагування Telegram/Viber/WhatsApp/Instagram/телефону/email — використовуються одночасно в модалці "Контакти", кнопці "Зв'язатися" в хіро й іконках у футері (все з одного джерела, оновлюється миттєво).
- **Списки** (статті/відгуки/проєкти): пагінація (20 записів/сторінку), пошук за заголовком/текстом, фільтр статусу, діапазон дат; весь рядок клікабельний для переходу в редагування, видалення — окрема кнопка з `confirm()`.
- **Підказки під полями** — короткий пояснювальний текст під кожним полем форм (що це і де використовується).

## 9. Публічна частина — функціонал

- Списки статей кожного типу з пагінацією (12/сторінку), фільтром за тегом (чипси) і пошуком за заголовком/тегом одночасно — сторінки пагінації зберігають активні фільтри.
- Сторінки статей показують теги-посилання назад до відфільтрованого списку.
- Банер-каруселька на головній: усі оголошення з позначкою `is_banner=1` (максимум 10), активні за датами, автопрокрутка + точки навігації.
- Проєкти: список з пагінацією (12/сторінку) + сторінка проєкту (H1-назва → необов'язковий лічильник-результат → фото/опис/CTA).
- Лічильник на головній: підтягується з одного опублікованого проєкту, позначеного "показувати на головній" — якщо такого немає (або проєкт чернетка), блок лічильника ховається повністю.
- Хіро-фото на головній редагується через адмінку (`/admin/home`), з фолбеком на оригінальне фото дизайну, якщо не задано.
- Каруселька відгуків на головній обмежена 12 записами (повний список — в адмінці).
- Контактна модалка + кнопка "Зв'язатися" в хіро + іконки контактів у футері — усі з одного джерела даних (`site_settings`).

## 10. SEO (реалізовано)

- Meta description, Open Graph, Twitter Card — автоматично з excerpt/фото кожного запису (запасний варіант — початок тіла статті, якщо excerpt порожній).
- `canonical` — обчислюється автоматично для кожної сторінки.
- JSON-LD структуровані дані: `BlogPosting`/`Article` для статей, `Organization` для головної, `CreativeWork` для проєктів.
- `noindex` автоматично для всієї адмінки.
- `/sitemap.xml` і `/robots.txt` — генеруються динамічно з реальних опублікованих записів, без ручного оновлення.
- По одному H1 на кожній публічній сторінці; змістовні `alt`-атрибути зображень.
- Усе оновлюється миттєво при редагуванні контенту — нічого не кешується.

## 12. Безпека — статус

**Реалізовано:**
- `require_admin()` на кожній мутуючій дії, не лише при вході.
- CSRF-токени на всіх формах (`hash_equals`, одноразовий `state` на OAuth-флоу теж).
- PDO з підготовленими запитами всюди.
- `htmlspecialchars` на весь виведений контент; HTML з Quill проходить через власний вайтлист-санітайзер (блокує `javascript:`, `data:`, зовнішні джерела фото).
- Завантаження фото валідується за реальним вмістом файлу (`getimagesize`, не за розширенням/MIME-заголовком), випадкові імена файлів, ліміт 5 МБ.
- httpOnly + secure + SameSite cookie, `session_regenerate_id()` після логіну/виходу.
- **Сесія адміна завершується сама** (30 хв бездіяльності / 8 год абсолютний ліміт).
- `/dev-login` гейтиться і `SERVER_NAME`, і `REMOTE_ADDR` (захист від підміни заголовка `Host`).
- `.htaccess` (`Require all denied`) на `app/`, `db/`, `config/`, `tmp/` — захист, якщо document root колись вкаже на корінь проєкту, а не на `public/`.
- `Options -Indexes` — вимкнено лістинг директорій (зокрема `/uploads/articles/`).
- Базові security-заголовки: `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy` (`public/.htaccess`).
- Лог спроб входу (`admin_login_log`).
- `noindex` на адмінці.
- `display_errors` вимкнено поза `IS_LOCAL` (деталі помилок не витікають на прод, пишуться в лог).

**Ще не реалізовано:**
- Content-Security-Policy (CSP) і HSTS — CSP поки не додавали свідомо: адмінка рясніє inline `<script>`/`style=""`, тож строгий CSP зараз її зламає, треба спершу винести інлайн-код в окремі файли.
- Rate limiting на `/login` — таблиця `admin_login_log` вже має все потрібне (email, ip, час), лишилось порахувати спроби й заблокувати.
- Стиснення/оптимізація завантажених зображень на сервері.
- Перевірка підпису GitHub webhook (деплой ще не налаштований).
- Регулярні автоматичні бекапи БД + `public/uploads`.
- Ролі з обмеженими правами (редактор без права видалення) — відкладено.

## 13. Відкриті питання

1. Реєстрація застосунку в Google Cloud Console (реальні Client ID/Secret) — зараз `/dev-login` як локальна заглушка.
2. Чи потрібні ролі з обмеженими правами (редактор без видалення) — відкладено до реальної потреби.
3. Перенесення 3 існуючих WordPress-сайтів з VIP30 на HQ10 — окрема задача.
4. **Код жодного разу не закомічений у git** (лише початковий комміт з `README.md`) — немає версійної історії поточного стану. Треба вирішити: комітити все як є, чи спершу почистити історію тестових експериментів.

## 14. Наступні кроки

1. Реєстрація Google Cloud Console застосунку, заповнення `config.php` реальними Client ID/Secret, вимкнення `/dev-login` на проді, додавання реального email(ів) у `admin_whitelist` (зараз там лише тестова адреса).
2. Rate limiting на логін, CSP (після винесення inline-скриптів адмінки), HSTS.
3. Стиснення зображень при завантаженні.
4. Налаштування деплою з GitHub (webhook з перевіркою підпису).
5. Перенесення сайту на прод-хостинг (див. `README.md` → розділ "Перенесення сайту").
6. Перенесення 3 існуючих сайтів з VIP30 на HQ10 (окрема задача).
7. Перший коміт поточного коду в git.
